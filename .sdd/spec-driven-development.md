# SDD Toolkit — Implementation Spec

## Overview

This document specifies the implementation of the `dotai sdd` subcommand — a Spec-Driven Development toolkit that integrates into the existing `dotai` CLI. It defines the command interface, directory structure, generated file contents, skill definitions, and README instructions that the AI agent must produce when the command runs.

---

## Context: How dotai Works

`dotai` is an npm CLI tool that manages AI coding tool configurations from a single `.ai/` source-of-truth directory. It symlinks `.ai/` subdirectories into the config directories each AI tool expects (`.claude/`, `.gemini/`, `.github/`, etc.), so editing one file keeps every tool in sync. `dotai` also maintains a persistent knowledge layer at `.ai/knowledge/` that gives every agent codebase context across sessions.

The `dotai init` command must be run before any subcommand. It creates the `.ai/` directory and sets up symlinks. The `sdd` subcommand builds on top of this initialized state.

---

## Command: `dotai sdd init`

### Prerequisite check

Before doing anything, the command must verify that `dotai` has already been initialized by checking that the `.ai/` directory exists in the project root. If it does not exist, the command must exit with a clear error:

```
Error: dotai is not initialized. Run `dotai init` first.
```

### What it does

When the prerequisite is satisfied, `dotai sdd init` performs the following in order:

1. Creates the `.ai/skills/` directory if it does not already exist.
2. Writes all 8 skill files into `.ai/skills/` (one subdirectory per skill, each containing a `SKILL.md`).
3. Creates the `.ai/sdd/` directory.
4. Writes a `README.md` into `.ai/sdd/` explaining the SDD workflow and how to use it.
5. Creates a template feature directory at `.ai/sdd/_template-feature/` pre-populated with a starter `idea.md` and empty placeholder files matching the expected structure.
6. Appends an instruction to the existing `.ai/AI.md` file (generated by `dotai init`) telling the agent to read `.ai/sdd/README.md` at the start of every session.

If `dotai sdd init` has already been run (i.e., `.ai/sdd/` already exists), the command should warn the user and exit without overwriting anything:

```
Warning: SDD toolkit already initialized. Nothing was changed.
```

### File naming convention

Task, plan, and evaluation files are always prefixed with a zero-padded two-digit index matching the task order: `01_`, `02_`, `03_`, etc. The title slug uses lowercase snake_case derived from the task title. The three files for a given task always share the same prefix and slug:

- `01_user_authentication.task.md`
- `01_user_authentication.plan.md`
- `01_user_authentication.evaluation.md`

## The 8-Phase SDD Workflow

This is the core workflow that the README and skills are built around. Every phase maps to one or more skill files.

### Phase 1 — Initiate

**Who:** The developer (human only, no agent).

**What:** The developer creates a new feature directory inside `.ai/sdd/` and writes `idea.md` by hand. This is a free-form prompt describing what they want to build, why it matters, and any constraints or goals already known. No structure is enforced — the goal is to capture raw intent clearly.

**Output:** `.ai/sdd/<feature-name>/idea.md`

**Skill used:** None. This phase is entirely human.

---

### Phase 2 — Specify

**Who:** Requirements Agent (AI).

**What:** The agent reads `idea.md` and generates a detailed `requirements.md` covering functional requirements, user flows, edge cases, and non-functional concerns. The agent must not invent scope — only formalize and expand what the developer already implied.

**Skill used:** `requirement-generation-skill`

**Output:** `.ai/sdd/<feature-name>/requirements.md`

**Human review checkpoint.** The developer must read `requirements.md` before the pipeline continues. This is the most critical review in the entire flow — a misaligned requirements document will silently corrupt every downstream phase. The developer should correct, add, or remove anything that does not match their intent and re-run the agent if significant changes are needed.

---

### Phase 3 — Decompose

**Who:** Decomposer Agent (AI).

**What:** The agent reads the approved `requirements.md` and breaks the work into discrete, implementable tasks. Each task is saved as its own file in the `tasks/` directory, named with the zero-padded index convention. Tasks must be scoped small enough for a coding agent to execute independently. The agent must define a clear order that respects dependencies between tasks.

**Skill used:** `task-decompose-skill`

**Output:** `.ai/sdd/<feature-name>/tasks/01_<slug>.task.md`, `02_<slug>.task.md`, ...

**Human review checkpoint.** Task order and granularity directly determine how implementation unfolds. If a task is too broad, the coding agent will make assumptions. If tasks are out of order, dependencies break. The developer must review all task files, reorder or rewrite as needed, and confirm the list is complete before moving to Phase 4.

---

### Phase 4 — Plan

**Who:** Planner Agent (AI). Two separate skills run in this phase.

**What:** For each task file in `tasks/`, the agent produces two corresponding output files:

- A **plan file** in `plans/` (`01_<slug>.plan.md`) — specifying exactly how to implement the task: which files to create or modify, which patterns and APIs to use, and the step-by-step implementation approach.
- An **evaluation file** in `evaluation/` (`01_<slug>.evaluation.md`) — defining the acceptance criteria for that task: what correct, complete output looks like, what edge cases must be handled, and how the evaluator should verify correctness.

**Skills used:** `plan-generation-skill` (for plan files), `evaluation-generation-skill` (for evaluation files).

**Output:**
- `.ai/sdd/<feature-name>/plans/01_<slug>.plan.md`, `02_<slug>.plan.md`, ...
- `.ai/sdd/<feature-name>/evaluation/01_<slug>.evaluation.md`, `02_<slug>.evaluation.md`, ...

**Human review checkpoint.** The evaluation files define the contract that Phase 6 enforces. Once accepted, they become the ground truth for automated quality checking. The developer must review both the plan files (for technical soundness) and the evaluation files (for correctness and completeness of acceptance criteria) before proceeding.

---

### Phase 5 — Implement

**Who:** Coding Agent (AI).

**What:** The agent works through the task list in order, one task at a time. For each task it reads the corresponding `.task.md` and `.plan.md` files and implements accordingly. It does not skip tasks or change their order. When a task is complete, it moves to the next. The agent runs autonomously through the full task list unless it encounters a blocker.

**Skill used:** `plan-implementation-skill`

**Output:** Working source code committed to the feature branch.

---

### Phase 6 — Evaluate

**Who:** Evaluator Agent (AI).

**What:** For each completed task, the agent reads the corresponding `.evaluation.md` file and checks the implementation against every acceptance criterion. It produces a pass or fail verdict per task with clear reasoning for any failure.

If any task fails, the loop returns to Phase 5. The Coding Agent receives the failure details and reattempts only the failing task. This loop repeats until all tasks pass. No code advances beyond Phase 6 until every evaluation criterion is satisfied.

**Skill used:** `evaluation-skill`

**Output:** Pass/fail evaluation result per task. Failed tasks trigger a Phase 5 retry.

---

### Phase 7 — Review

**Who:** Code Reviewer Agent (AI) + developer (human).

**What:** The agent reads the entire codebase holistically — not task by task, but as a complete system — and produces a review report. It looks for issues only visible at the system level: inconsistent patterns, redundant logic, security concerns, naming inconsistencies, and anything that passed per-task evaluation but looks wrong in context.

**Skill used:** `code-review-skill`

**Output:** A written code review report.

**Human review checkpoint (final gate).** The developer reads the review report and decides what must be fixed before shipping. No code should be merged or deployed without the developer having read this report and signed off. This is the last human gate in the pipeline.

---

### Phase 8 — Context-sync

**Who:** Context Agent (AI).

**What:** The agent updates the `.ai/knowledge/` directory — refreshing memory files, agent rules, and codebase documentation to reflect everything that was built and decided during this feature cycle. This ensures the next development session starts with full awareness of what exists, what patterns were established, and what conventions emerged.

**Skill used:** `knowledge-update-skill`

**Output:** Updated files in `.ai/knowledge/`.

## Skill File Contents

Each skill file (`SKILL.md`) is a structured prompt that tells the AI agent exactly what to do when that skill is invoked. The agent reads the skill file before taking action.

### `.ai/skills/requirement-generation-skill/SKILL.md`

```markdown
# Skill: Requirement Generation

## Purpose
Read the developer's raw idea from `idea.md` and produce a complete, structured `requirements.md` for the feature.

## Instructions
1. Read `.ai/sdd/<feature-name>/idea.md` in full before writing anything.
2. Do not invent scope. Only formalize and expand what the developer has already implied.
3. Produce `requirements.md` with the following sections:
   - **Overview** — one paragraph restating the goal in clear language.
   - **Functional Requirements** — numbered list of specific, testable capabilities the system must have.
   - **User Flows** — step-by-step descriptions of how a user moves through the feature.
   - **Edge Cases** — explicit list of boundary conditions and error states that must be handled.
   - **Non-Functional Requirements** — performance, security, accessibility, or compatibility constraints.
   - **Out of Scope** — explicit list of things this feature does NOT cover.
4. Every functional requirement must be testable. Avoid vague language like "the system should be fast" — write "the API must respond within 300ms under normal load" instead.
5. Save the output to `.ai/sdd/<feature-name>/requirements.md`.
6. After saving, notify the developer that human review is required before the pipeline continues.
```

---

### `.ai/skills/task-decompose-skill/SKILL.md`

```markdown
# Skill: Task Decomposition

## Purpose
Break an approved `requirements.md` into a set of discrete, implementable task files in the `tasks/` directory.

## Instructions
1. Read `.ai/sdd/<feature-name>/requirements.md` in full.
2. Identify every distinct unit of work required to fulfill the requirements.
3. For each task:
   - Assign a zero-padded index reflecting its execution order (01, 02, 03, ...).
   - Derive a short snake_case slug from the task title.
   - Write a task file named `<index>_<slug>.task.md` into `.ai/sdd/<feature-name>/tasks/`.
4. Each task file must contain:
   - **Task title**
   - **Goal** — one sentence describing what this task achieves.
   - **Depends on** — list of task indices this task must follow (empty if none).
   - **Input files** — existing files the coding agent must read to do this task.
   - **Output files** — files the coding agent must create or modify.
   - **Definition of done** — specific, observable criteria that confirm the task is complete.
5. Tasks must be ordered so all dependencies are satisfied before a task is attempted. No circular dependencies.
6. Tasks must be granular enough for a coding agent to execute autonomously without making architectural decisions.
7. After writing all task files, notify the developer that human review is required.
```

---

### `.ai/skills/plan-generation-skill/SKILL.md`

```markdown
# Skill: Plan Generation

## Purpose
For each task file in `tasks/`, produce a corresponding implementation plan file in `plans/`.

## Instructions
1. Read `.ai/sdd/<feature-name>/requirements.md` for full context.
2. For each file in `.ai/sdd/<feature-name>/tasks/`:
   a. Read the task file completely.
   b. Produce a plan file named with the same index and slug: `<index>_<slug>.plan.md`.
   c. Save it to `.ai/sdd/<feature-name>/plans/`.
3. Each plan file must contain:
   - **Task reference** — the task index and title this plan belongs to.
   - **Implementation steps** — ordered, numbered steps the coding agent must follow.
   - **Files to create** — exact file paths and a description of their purpose.
   - **Files to modify** — exact file paths, which sections to change, and what to change them to.
   - **Patterns and conventions** — specific coding patterns, naming conventions, or architectural rules to follow.
   - **APIs and interfaces** — any internal or external APIs the task must call or implement.
   - **Error handling** — how the implementation must handle failures and edge cases.
4. Plans must be detailed enough that a coding agent can follow them without making design decisions.
```

### `.ai/skills/evaluation-generation-skill/SKILL.md`

```markdown
# Skill: Evaluation Generation

## Purpose
For each task file in `tasks/`, produce a corresponding acceptance criteria file in `evaluation/`.

## Instructions
1. Read `.ai/sdd/<feature-name>/requirements.md` for full context.
2. For each file in `.ai/sdd/<feature-name>/tasks/`:
   a. Read the task file and its corresponding plan file in `plans/`.
   b. Produce an evaluation file named with the same index and slug: `<index>_<slug>.evaluation.md`.
   c. Save it to `.ai/sdd/<feature-name>/evaluation/`.
3. Each evaluation file must contain:
   - **Task reference** — the task index and title this evaluation belongs to.
   - **Acceptance criteria** — numbered list of specific, verifiable conditions that must all be true for the task to pass.
   - **Test scenarios** — concrete input/output examples that demonstrate each criterion.
   - **Edge case checks** — specific edge conditions the evaluator must verify.
   - **Files to inspect** — exact file paths the evaluator must examine.
   - **Failure examples** — examples of what a failing implementation looks like, to help the evaluator make accurate judgements.
4. Every criterion must be verifiable by reading code or running the application. Avoid subjective criteria.
5. After writing all evaluation files, notify the developer that human review is required before implementation begins.
```

### `.ai/skills/plan-implementation-skill/SKILL.md`

```markdown
# Skill: Plan Implementation

## Purpose
Execute the implementation plan for a single task by following its plan file exactly.

## Instructions
1. Identify the current task to implement. Always work on the lowest-indexed incomplete task.
2. Read the following files before writing any code:
   - `.ai/sdd/<feature-name>/tasks/<index>_<slug>.task.md`
   - `.ai/sdd/<feature-name>/plans/<index>_<slug>.plan.md`
   - `.ai/knowledge/` — for codebase context and established conventions.
3. Follow the implementation steps in the plan file in order. Do not skip steps.
4. Do not make design decisions not covered by the plan. If something is unclear, check `requirements.md` and `plans/` first. If still unclear, stop and ask the developer.
5. After completing the task, verify that all output files listed in the task file have been created or modified as specified.
6. Do not begin the next task automatically. Report completion and wait.
```

---

### `.ai/skills/evaluation-skill/SKILL.md`

```markdown
# Skill: Evaluation

## Purpose
Verify a completed task implementation against its acceptance criteria and produce a clear pass or fail verdict.

## Instructions
1. Identify the task to evaluate by its index.
2. Read the following files:
   - `.ai/sdd/<feature-name>/evaluation/<index>_<slug>.evaluation.md`
   - `.ai/sdd/<feature-name>/tasks/<index>_<slug>.task.md`
   - All files listed under "Files to inspect" in the evaluation file.
3. Check every acceptance criterion in the evaluation file. For each criterion, record:
   - Whether it passes or fails.
   - The specific evidence from the code that supports this verdict.
4. Produce an evaluation report with:
   - **Overall verdict** — PASS or FAIL.
   - **Criterion results** — the pass/fail verdict and evidence for each criterion.
   - **Failure details** — for any failing criterion, a precise description of what is wrong and what the correct implementation should look like.
5. If the overall verdict is FAIL, the Coding Agent must retry the task using the failure details as input. Do not advance to the next task until the verdict is PASS.
6. If the overall verdict is PASS, report completion and proceed to the next task's evaluation.
```

### `.ai/skills/code-review-skill/SKILL.md`

```markdown
# Skill: Code Review

## Purpose
Review the entire implementation of a completed feature holistically and produce a structured review report.

## Instructions
1. Read the following context files first:
   - `.ai/sdd/<feature-name>/requirements.md`
   - `.ai/sdd/<feature-name>/tasks/` — all task files.
   - `.ai/knowledge/` — for established project conventions.
2. Read the full implementation — every file created or modified during this feature cycle.
3. Review the code as a complete system, not task by task. Look specifically for:
   - **Consistency** — naming conventions, code style, and architectural patterns inconsistent with the rest of the codebase.
   - **Redundancy** — logic or utilities duplicated across files that should be shared.
   - **Security** — input validation gaps, insecure defaults, exposed secrets, or unsafe data handling.
   - **Error handling** — missing or insufficient handling of failure states.
   - **Test coverage** — missing tests for critical paths or edge cases.
   - **Correctness** — any behaviour that does not match the requirements even if it passed per-task evaluation.
4. Produce a review report with:
   - **Summary** — overall assessment in 2-3 sentences.
   - **Issues** — each issue listed with severity (critical / major / minor), the affected file and line range, a description of the problem, and a suggested fix.
   - **Positives** — notable things done well.
5. Do not auto-fix anything. The review report is for the developer to read and act on.
```

### `.ai/skills/knowledge-update-skill/SKILL.md`

```markdown
# Skill: Knowledge Update

## Purpose
Update the `.ai/knowledge/` directory to reflect everything built and decided during this feature cycle, so future sessions start with accurate codebase context.

## Instructions
1. Read the completed feature artifacts:
   - `.ai/sdd/<feature-name>/requirements.md`
   - `.ai/sdd/<feature-name>/tasks/` — all task files.
   - `.ai/sdd/<feature-name>/plans/` — all plan files.
2. Read the existing `.ai/knowledge/` directory to understand what is already documented.
3. Update or create knowledge files to reflect:
   - **New modules or components** — what they do, where they live, and how they are structured.
   - **New patterns or conventions** — coding patterns established during this feature that should be followed in future work.
   - **New APIs or interfaces** — internal APIs created, with their signatures and usage examples.
   - **Architectural decisions** — significant decisions made and the reasoning behind them.
   - **Known limitations** — anything intentionally deferred or left incomplete, and why.
4. Do not delete existing knowledge entries unless they are factually incorrect. Prefer adding or updating.
5. Keep knowledge files concise — they are reference material for agents, not documentation for humans.
```

## `.ai/sdd/README.md` Contents

This file is the primary human-readable guide. It explains the workflow, lists the slash commands for each phase, and tells both developers and AI agents how to use the toolkit.

```markdown
# SDD Toolkit

Spec-Driven Development (SDD) is a structured workflow where specifications drive implementation rather than emerging from it. Every piece of code produced by an AI agent in this project traces back to a human-reviewed specification.

This toolkit integrates with `dotai` and lives at `.ai/sdd/`. It provides skills, templates, and a defined workflow that takes a raw feature idea through 8 phases to a reviewed, context-synced implementation.

---

## Workflow Overview

| Phase | Name | Who | Skill | Output |
|-------|------|-----|-------|--------|
| 1 | Initiate | Human | — | `idea.md` |
| 2 | Specify | Agent | `requirement-generation-skill` | `requirements.md` |
| 3 | Decompose | Agent | `task-decompose-skill` | `tasks/*.task.md` |
| 4 | Plan | Agent | `plan-generation-skill` + `evaluation-generation-skill` | `plans/*.plan.md` + `evaluation/*.evaluation.md` |
| 5 | Implement | Agent | `plan-implementation-skill` | Source code |
| 6 | Evaluate | Agent | `evaluation-skill` | Pass/fail per task |
| 7 | Review | Agent + Human | `code-review-skill` | Review report |
| 8 | Context-sync | Agent | `knowledge-update-skill` | Updated `.ai/knowledge/` |

Human review is required after Phase 2, Phase 3, Phase 4, and Phase 7.

---

## Starting a New Feature

### Step 1 — Create the feature directory

Copy the template feature directory and rename it to your feature:

```bash
cp -r .ai/sdd/_template-feature .ai/sdd/my-feature-name
```

### Step 2 — Write your idea

Open `.ai/sdd/my-feature-name/idea.md` and fill it in by hand. This is the only file you write yourself. Be as clear and specific as you can about what you want to build and why.

### Step 3 — Run the pipeline

Open your AI coding agent and run the phases in order using the skills below.

---

## How to Use Each Skill

### Phase 2 — Specify
```
Use the skill at .ai/skills/requirement-generation-skill/SKILL.md to generate requirements for .ai/sdd/my-feature-name/
```

**Review** `requirements.md` before continuing. Correct anything that does not match your intent.

### Phase 3 — Decompose
```
Use the skill at .ai/skills/task-decompose-skill/SKILL.md to decompose .ai/sdd/my-feature-name/requirements.md into tasks
```

**Review** all files in `tasks/` before continuing. Check that the order is correct and every requirement is covered.

### Phase 4 — Plan
```
Use the skill at .ai/skills/plan-generation-skill/SKILL.md to generate plans for all tasks in .ai/sdd/my-feature-name/
```
```
Use the skill at .ai/skills/evaluation-generation-skill/SKILL.md to generate evaluation criteria for all tasks in .ai/sdd/my-feature-name/
```

**Review** both `plans/` and `evaluation/` before continuing. The evaluation files are the contract for Phase 6.

### Phase 5 — Implement
```
Use the skill at .ai/skills/plan-implementation-skill/SKILL.md to implement the next incomplete task in .ai/sdd/my-feature-name/
```

Repeat for each task. Do not run Phase 6 until a task is complete.

### Phase 6 — Evaluate
```
Use the skill at .ai/skills/evaluation-skill/SKILL.md to evaluate task 01 in .ai/sdd/my-feature-name/
```

If the verdict is FAIL, the agent will retry Phase 5 for that task. If the verdict is PASS, move to the next task.

### Phase 7 — Review
```
Use the skill at .ai/skills/code-review-skill/SKILL.md to review the full implementation of .ai/sdd/my-feature-name/
```

**Read the review report** and decide what to fix before merging.

### Phase 8 — Context-sync
```
Use the skill at .ai/skills/knowledge-update-skill/SKILL.md after completing .ai/sdd/my-feature-name/
```

Run this at the end of every feature cycle without exception.

---

## Rules for AI Agents

- Always read the relevant skill file before starting any phase.
- Never skip a phase or change the order.
- Never proceed past a human review checkpoint without explicit developer confirmation.
- Always read `.ai/knowledge/` before starting Phase 5 to understand existing codebase conventions.
- If a task or plan is ambiguous, stop and ask the developer. Do not make assumptions.
- The evaluation files in `evaluation/` are the source of truth for correctness. They override your own judgement.

---

## File Naming Convention

Task, plan, and evaluation files share the same naming pattern:

```
<zero-padded-index>_<snake_case_title>.<type>.md
```

Examples:
- `01_user_authentication.task.md`
- `01_user_authentication.plan.md`
- `01_user_authentication.evaluation.md`

The three files for a given task always have the same index and slug.
```

## SDD Toolkit

This project uses a Spec-Driven Development workflow managed by the `dotai sdd` toolkit.

Read `.ai/sdd/README.md` at the start of every session before writing any code for a feature that has an entry in `.ai/sdd/`.

## Implementation Notes for the Coding Agent

- All file writes must be idempotent — running `dotai sdd init` twice must not corrupt existing files.
- The `_template-feature/` directory name is prefixed with an underscore to visually distinguish it from real feature directories in file explorers.
- `.gitkeep` files in `tasks/`, `plans/`, and `evaluation/` subdirectories of the template ensure the directories are tracked by git even when empty.
- The command must log each created file to stdout so the developer can see exactly what was generated.
- No external dependencies beyond what `dotai` already uses should be introduced for this subcommand.
